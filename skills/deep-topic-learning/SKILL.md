# 深度技术主题学习（Deep Topic Learning）

把「难学的技术」拆成可重复的工作流：**先建地图 → 拆模块跑通 → 读源码对齐认知 → 用自己的话总结并接受纠错 → 把坑写成疑惑档案**。适用于任意领域（后端、Agent、向量库、编译器、业务系统），只要材料复杂、层次多、易迷路。

## 何时启用

- 用户要**系统理解**某块能力，而不是抄一段能跑的代码就结束。
- 文档厚、概念多、有多套存储/模型/层时。
- 用户明确要 **沉淀**：README、分章笔记、`.ipynb`、团队分享。
- 学习过程中**反复踩坑**（配置、版本、端口、API 变更），需要可追溯记录。

## 核心工作流（五段式）

按顺序推进；任一阶段发现「地图错了」就回到上一阶段修正，不要硬扛。

### 1. 建地图（Mental model first）

**目标**：用一张「能指路」的鸟瞰图，而不是细节堆砌。

**做法**：

- 先答三个问题：**输入是什么、输出是什么、谁调用谁**（分层、数据流）。
- 画简单示意图：箭头或 ASCII 即可（`A → B → C`）。
- 做一张**对比总览表**（维度 × 方案 × 为什么），强制区分「相似但不同」的概念。
- 标明**权威信息源**：官方文档 vs 本仓库源码——冲突时以可验证的为准。

**产出物**：顶层 `README.md` 的一节「架构与数据流」，或独立 `00_overview.md`。

### 2. 拆模块跑通（Vertical slices）

**目标**：每个模块**独立可运行**，降低认知负载。

**做法**：

- 按**用户任务路径**切模块（例如：写入路径一条、检索路径一条），而不是按文件目录机械切分。
- 每模块：**最小可运行示例** + 预期现象（打印关键字段、日志、状态码）。
- 复杂交互用 Notebook 时：每个主题一个文件；先 Markdown 讲「这一章要验证什么」，再 Code。

**产出物**：分文件教程、或一组 `01_*.ipynb`…；每章末尾预留「个人总结 / 学习疑惑」锚点（见第 5 节）。

### 3. 读源码对齐认知（Ground truth）

**目标**：消除「文档这么说但实际不是那么回事」的偏差。

**做法**：

- 对**争议点、性能声称、数据结构选择**（如 List vs Heap），直接 **open 实现文件**，跟调用链走一遍。
- 记录：**哪个函数**、**哪几个分支**、**环境变量/配置**如何改变行为。
- 若第三方闭源或太大：以**公开 API + 行为实验**（最小复现脚本）对齐，并注明推断边界。

**产出物**：在对应章节加一小节「内部实现要点」或脚注链接到源码路径（仓库内或 `site-packages` 路径写清楚，便于他人复查）。

### 4. 用自己的话总结并接受纠错（Active recall）

**目标**：把「看过」变成「能讲清楚且经得住追问」。

**做法**：

- 请学习者写**短总结**（数据结构、关键步骤、公式/评分逻辑）。
- 协作者（或 AI）只做两件事：**指出事实错误**、**补边界条件**（什么情况下不成立）。
- 修正后的总结**写回文档**，避免只留在聊天里。

**产出物**：每章「个人总结」段落（可迭代追加，标注日期或版本更好）。

### 5. 疑惑档案（Incident / FAQ log）

**目标**：同一类坑只踩一次；团队后来者能检索。

**做法**：

每条记录尽量统一骨架（可复制 `references/artifact-templates.md`）：

1. **现象**（报错原文、日志关键行、HTTP 状态码）
2. **误判**（曾经以为的原因）
3. **根因**（配置 / 版本 / 端口 / API 变更 / 误解）
4. **验证**（一条命令或最小脚本如何证明）
5. **正确做法**（改什么、命令是什么）
6. **原则**（例如：优先锁依赖版本，避免改 `site-packages`）

**产出物**：每章「学习疑惑」或项目级 `TROUBLESHOOTING.md`；标题和关键词要可搜索（产品名、版本号、错误码）。

## 文档分层建议（便于拓展）

| 层级 | 用途 | 典型内容 |
|------|------|----------|
| 地图层 | 30 秒～2 分钟建立方向 | 架构图、对比表、术语表 |
| 模块层 | 跟练与验证 | 分章 md/ipynb、可运行 cell |
| 真相层 | 纠偏 | 源码引用、配置表 |
| 沉淀层 | 长期资产 | 个人总结、疑惑档案 |

新增主题时：**先扩地图表**，再**加新模块文件**，不要把所有东西堆进一个 README。

## 与工程习惯的衔接

- **依赖/API 不兼容**：默认走 **版本约束**（`pyproject.toml` / `uv.lock` / `requirements.txt`），在疑惑档案里写清上下限；改第三方安装目录里的源码是最后手段且需注明「易被覆盖」。
- **环境变量**：在疑惑档案中写「读到的值 vs 期望的值」，避免 `load_dotenv` 与 Shell 导出不一致的隐性坑。
- **「连接成功」类日志**：提醒区分「客户端对象创建」与「真正 RPC/HTTP 探测成功」。

## 对 AI 协作者的执行提示

当用户在学习模式下工作时：

1. **先确认地图**是否已有；没有则与用户共建一版再下钻。
2. **拆任务**时优先独立可验证步骤；避免一次改十个文件却无处对照。
3. **引用源码**时用仓库路径 + 函数名，便于用户跳转。
4. **用户总结后**必须做事实核对，再建议写入「个人总结」。
5. **任何排错结论**顺手整理成疑惑档案条目，并问用户是否写入指定章节。

## 进阶：从个人学习到可分享教程

- 统一术语表（中英对照可选）。
- 每章「读完应能回答」列 2～3 个自检问题。
- 重大勘误在地图层同步更新，避免模块层与地图矛盾。
